# Security findings

## S1 — Privileged role bypass in policy evaluation

**Severity:** High
**Location:** `src/auth/policy.ts:179-200`

```ts
evaluate(context: PolicyContext): PolicyVerdict {
  const isPrivileged =
    context.user_role === 'owner' || context.user_role === 'admin';

  if (isPrivileged && this.privilegedBypassBudgets) {
    return { ...this.defaultVerdict, allowed: true };
  }

  if (isPrivileged) {
    let verdict: PolicyVerdict = { ...this.defaultVerdict, allowed: true };
    for (const rule of this.rules) {
      if (!this.ruleMatches(rule, context)) continue;
      if (rule.cost_ceiling_usd !== undefined) {
        verdict = { ...verdict, cost_ceiling_usd: rule.cost_ceiling_usd };
      }
    }
    return verdict;
  }
  ...
}
```

**Impact.** Any context evaluated with `user_role === 'owner' | 'admin'` short-circuits the entire policy pipeline. With the legacy `privileged_bypass_budgets` flag enabled, *all* per-rule constraints — allowlists, model-tier restrictions, denials, confirmation requirements — are silently skipped. Even with that flag off, only `cost_ceiling_usd` from matching rules is honored; explicit denials, executor allowlists, and `requires_confirmation` lists are ignored. A compromised admin Telegram token (or a misclassified group-admin) becomes an unconstrained spend and action vector against the orchestrator.

This is **explicitly known** — the design note at lines 6-50 lays out the proposed `applies_to_privileged` hardening in detail. It just hasn't shipped.

**Fix.** Implement the proposal in the design comment:
1. Add `applies_to_privileged?: boolean` (default `false`) to `PolicyRule`.
2. Apply matching rules to privileged users when that flag is set.
3. Always enforce `cost_ceiling_usd` from the *default verdict*, regardless of role — not only from rules.
4. Default `privileged_bypass_budgets` to `false`; require an explicit opt-in for legacy behavior.

## S2 — Unsigned-webhook escape hatch tied only to `NODE_ENV`

**Severity:** Medium
**Location:** `src/webhooks/gateway.ts:235-249`

```ts
private verifyRequest(adapter: ChannelAdapter, req: Request): boolean {
  if (typeof adapter.verifyWebhookSignature === 'function') {
    const rawBody = (req as Request & { rawBody?: Buffer }).rawBody;
    return adapter.verifyWebhookSignature(
      req.headers as Record<string, string | string[] | undefined>,
      rawBody
    );
  }
  return this.isUnsignedAllowed();
}

private isUnsignedAllowed(): boolean {
  if (process.env['NODE_ENV'] === 'production') return false;
  return process.env['ALDUIN_ALLOW_UNSIGNED'] === '1';
}
```

**Impact.** Two ways to receive forged webhook traffic in non-production deployments:

1. An adapter with no `verifyWebhookSignature` method silently falls through to `isUnsignedAllowed()`. Any future channel adapter that forgets to implement the method gets *no* signature checking.
2. If `NODE_ENV` is missing, blank, or set to anything other than the exact string `'production'` (e.g., a typo, a staging environment using `staging`, a container that forgot to set it), `ALDUIN_ALLOW_UNSIGNED=1` enables unsigned ingest.

The Telegram path itself (`channels/telegram/index.ts:162-185`) is implemented correctly with `crypto.timingSafeEqual` and fail-closed on missing secret — that part is fine. The risk is the gateway-level fallback being looser than it looks.

**Fix.**
- Make `verifyWebhookSignature` mandatory on the `ChannelAdapter` interface (compile-time enforced) — drop the `typeof === 'function'` check.
- Replace the `NODE_ENV !== 'production'` guard with an explicit allowlist: require an environment that *is* known to be a dev environment, e.g. `ALDUIN_ENV === 'development'`. Default-deny when `ALDUIN_ENV` is unset.
- Log a loud warning at startup whenever `ALDUIN_ALLOW_UNSIGNED=1` is observed.

## S3 — Plugin entry modules loaded without signature verification

**Severity:** Medium
**Location:** `src/plugins/loader.ts`, `loadEntryModule`

```ts
async function loadEntryModule(
  entryPath: string,
  manifest: AlduinPluginManifest,
): Promise<PluginExports> {
  // TODO Phase 4: signed-manifest verification before import
  const entryUrl = pathToFileURL(entryPath).href;
  const mod = await import(entryUrl) as Record<string, unknown>;
  ...
}
```

**Impact.** Plugins are dynamically `import()`ed from disk with no integrity check. Any path that can write to the plugin directory — a compromised dependency, a malicious npm postinstall, an adversary with filesystem access, an unintended `git pull` of a tampered branch — gets RCE inside the orchestrator process. That process holds the master vault key in memory (`src/secrets/vault.ts:104`), so a compromised plugin directly exfiltrates every stored OAuth token and connector credential.

The `TODO Phase 4` comment shows the team is aware. The point of flagging it here is that the impact is high (full credential vault compromise) even though the surface is narrower than network-facing code.

**Fix.**
- Hash each plugin's entry file at install time, store the hash in a vault-protected manifest table, verify before each `import()`.
- For third-party plugins, require Ed25519 signature over the manifest hash plus a curated public-key allowlist.
- Reject load on hash mismatch; never auto-rebuild trust on file change.

## S4 — Skill "sandbox" naming is misleading; isolation is advisory

**Severity:** Info (with caller-trust caveat)
**Location:** `src/skills/sandbox.ts:1-202`

The header comment is admirably honest:

> ⚠ NOT A SECURITY BOUNDARY. The historical name "sandbox" is retained for code-identifier stability only … Treat this as a soft isolation layer for well-behaved but potentially buggy skill code, not as a barrier against untrusted input or malicious plugins.

And the implementation matches the disclaimer — `worker_threads` share the parent process memory; the require-blocklist (`fs`, `http`, etc.) can be bypassed by any native-code exploit; `--max-old-space-size` is advisory.

**Why it's listed at all.** The file name and the exported `runInSandbox` function will still get used by people who didn't read the comment. Two concrete risks:

1. The earlier security pass flagged the `JSON.stringify(`'use strict';\n${code}\n`)` interpolation as injectable. After verification this is **not** an injection — `JSON.stringify` produces a valid JS string literal in every case. The risk is *not* injection, it's **trust**: the worker compiles whatever JavaScript it receives via `new AsyncFunction(...)`. Anything that can supply `code` to `runInSandbox` already has full process access via the worker.
2. Combined with **S3** (unsigned plugin loading) the chain is: tampered plugin → calls `runInSandbox` with crafted code → escapes "soft" isolation → reads vault.

**Fix / hardening recommendations.**
- Rename the symbol or the export (`runInWorker`, `runUntrustedJs`) so callers can't pretend to themselves it's a sandbox.
- Document the `code` parameter as *trusted operator-supplied JavaScript only* in the JSDoc, and add a runtime check (e.g., reject if `code` originated from an LLM tool result or a network input).
- Track the `isolated-vm`/separate-process work item — it's the right long-term answer.

## S5 — Memory redactor pattern coverage is narrow

**Severity:** Low
**Location:** `src/memory/redactor.ts:25-39`

The redactor catches OpenAI keys, Anthropic keys, GitHub PATs, JWTs, AWS access key IDs, and (opt-in) emails / NA phone numbers. It does **not** catch:

- Slack tokens (`xoxb-`, `xoxp-`, `xoxa-`, `xoxs-`)
- Google API keys (`AIza[0-9A-Za-z\-_]{35}`)
- Stripe keys (`sk_live_`, `pk_live_`, `rk_live_`)
- Generic `Authorization: Bearer ...` headers
- AWS *secret* access keys (the 40-char companion to `AKIA...`)
- Telegram bot tokens (`\d{8,10}:[A-Za-z0-9_-]{35}`)
- GitHub fine-grained PATs (`github_pat_` prefix)
- Private keys (`-----BEGIN (RSA |EC |OPENSSH )?PRIVATE KEY-----`)

Since redaction only fires on hot→warm/cold promotion, leaks of these credentials persist in cold storage indefinitely.

**Fix.** Extend `SECRET_PATTERNS` with the entries above. Consider adopting a maintained ruleset (e.g., the patterns from `gitleaks` or `truffleHog`) and importing it as data rather than maintaining hand-rolled regexes.

# Bug findings

## B1 — Timeout handle leak in MCP tool host (success path)

**Severity:** High
**Location:** `src/plugins/mcp-host.ts:195-212`

```ts
let timeoutHandle: ReturnType<typeof setTimeout> | undefined;
let timedOut = false;
const timeoutPromise = new Promise<never>((_, reject) => {
  timeoutHandle = setTimeout(() => {
    timedOut = true;
    controller.abort();
    reject(new Error(`Tool "${call.name}" timed out after ${timeoutMs}ms`));
  }, timeoutMs);
});

let result: ToolResult;
try {
  result = await Promise.race([
    entry.plugin.invoke(call.name, call.arguments, ctxWithSignal),
    timeoutPromise,
  ]);
} catch (error) {
  if (timeoutHandle) clearTimeout(timeoutHandle);
  ...
}
```

**Bug.** `clearTimeout(timeoutHandle)` is called only in the `catch` block. When `entry.plugin.invoke(...)` resolves *successfully* (the common case), the timer is never cleared. It fires later, calls `controller.abort()` on the now-completed call's `AbortController`, and rejects a `Promise<never>` that nothing is listening to.

**Symptoms during normal usage.**
- Every successful tool invocation pins one closure (`reject`, `controller`, `call.name`) in memory until `timeoutMs` elapses (default 30s — see `invokeTimeoutMs`). At sustained tool throughput this creates a large floating set.
- The Node event loop will not exit cleanly while these timers are pending; the orchestrator's `close()` won't return promptly.
- On graceful shutdown, the still-armed timers will fire after the `AbortController` is logically dead, possibly emitting "abort after disposal" log noise from cooperating plugins.

**Fix.** Use `finally`:

```ts
try {
  result = await Promise.race([
    entry.plugin.invoke(call.name, call.arguments, ctxWithSignal),
    timeoutPromise,
  ]);
} catch (error) { ... }
finally {
  if (timeoutHandle) clearTimeout(timeoutHandle);
}
```

## B2 — Timeout handle leak in sub-orchestration dispatch

**Severity:** High
**Location:** `src/executor/dispatch.ts:221-231`

```ts
const timeoutPromise = new Promise<never>((_, reject) => {
  setTimeout(
    () => reject(new Error(`Child orchestration timed out after ${orch.parent_timeout_remaining_ms}ms`)),
    orch.parent_timeout_remaining_ms,
  );
});

const { response, trace: childTrace } = await Promise.race([
  childPromise,
  timeoutPromise,
]);
```

**Bug.** The `setTimeout` handle is never assigned to a variable, never cleared, and never canceled. Every successful child orchestration leaves a timer that will fire at `parent_timeout_remaining_ms` later, rejecting an unobserved promise.

**Symptoms.** Identical to B1 but worse magnitude — `parent_timeout_remaining_ms` is the *parent* timeout, often minutes, and recursive workflows multiply the leak. A 5-deep recursion at 5-minute parent timeout pins ~5 closures per branch for ~5 minutes after each branch completes.

**Fix.** Capture the handle and clear in `finally`, or wrap in a small helper:

```ts
let handle: ReturnType<typeof setTimeout> | undefined;
const timeoutPromise = new Promise<never>((_, reject) => {
  handle = setTimeout(
    () => reject(new Error(`Child orchestration timed out after ${orch.parent_timeout_remaining_ms}ms`)),
    orch.parent_timeout_remaining_ms,
  );
});
try {
  return await Promise.race([childPromise, timeoutPromise]);
} finally {
  if (handle) clearTimeout(handle);
}
```

## B3 — Timeout handle leak in standard executor dispatch (`createTimeout`)

**Severity:** High
**Location:** `src/executor/dispatch.ts:366-369` and the `createTimeout` helper

```ts
try {
  result = await Promise.race([
    provider.complete(request),
    this.createTimeout(task.timeout_ms),
  ]);
} catch { ... }
```

`createTimeout` (per the bug-hunt agent's read) returns a `Promise` whose `setTimeout` handle is never exposed, so the caller cannot clear it.

**Bug & symptoms.** Same class as B1/B2: every successful LLM completion leaves a timer pinning a closure for `task.timeout_ms`. Because this fires on the **hot path of every executor call**, the leak rate is *one timer per LLM completion*. A long-running agent issuing thousands of completions accumulates thousands of pending timers; aside from heap pressure, it prevents the Node process from exiting cleanly when the operator hits Ctrl-C.

**Fix.** Refactor `createTimeout` to accept an `AbortSignal`, or rewrite the call site to manage the handle directly:

```ts
private async raceWithTimeout<T>(p: Promise<T>, ms: number): Promise<T> {
  let handle: ReturnType<typeof setTimeout> | undefined;
  const timer = new Promise<never>((_, reject) => {
    handle = setTimeout(() => reject(new Error(`Timeout after ${ms}ms`)), ms);
  });
  try {
    return await Promise.race([p, timer]);
  } finally {
    if (handle) clearTimeout(handle);
  }
}
```

The same helper should replace B1 and B2 — collapse all three call sites onto it.

## B4 — Stream-consumer final-flush condition is a tautology

**Severity:** Low
**Location:** `src/renderer/stream-consumer.ts:111`

```ts
// Final partial flush (if there's unsent content)
if (content && Date.now() - lastPartialAt > 0) {
  callbacks.onPartial(content);
}
```

**Bug.** `Date.now() - lastPartialAt > 0` is true whenever `lastPartialAt` is in the past, which it always is (it's set during stream processing or initialized to `0`). The intent — judging by the comment — is "flush only if we still have unsent content since the last throttled partial." The condition does not encode that.

**Practical effect.** When the throttled partial happened to fire on the *very last* chunk of a stream, this will re-emit `onPartial(content)` with the same text (or a near-duplicate), causing UI duplication or one extra subscriber notification. Not data corruption, but visible jitter on slow streams.

**Fix.** Track whether content has changed since the last partial:

```ts
// In the streaming loop, when calling onPartial:
lastPartialContent = content;

// At the end:
if (content && content !== lastPartialContent) {
  callbacks.onPartial(content);
}
```

Or, if the goal is "always flush at end as long as we ever emitted any partial":

```ts
if (content && lastPartialAt > 0) {
  callbacks.onPartial(content);
}
```

# What was checked but found clean

These came up as suspicious during the audit and were verified to be implemented correctly — listing them so future audits don't re-flag:

- **Telegram webhook signature verification** (`src/channels/telegram/index.ts:162-185`) — uses `crypto.timingSafeEqual`, length-checks before compare, fails closed on missing secret. Correct.
- **OAuth flow** (`src/connectors/oauth.ts`) — PKCE S256, one-time state with TTL sweep + map cap, vault transactions for token storage, AbortSignal timeouts on token endpoint, no client-credential leakage in URLs.
- **Vault crypto** (`src/secrets/vault.ts:14-105`) — AES-256-GCM with 12-byte IV per write, scrypt with hardened params (N=2³¹⁷, maxmem 256 MiB), per-install random salt, legacy-salt migration in transaction, file mode 0o600 / dir 0o700.
- **Webhook gateway hardening** (`src/webhooks/gateway.ts:60-155`) — binds 127.0.0.1 by default, refuses `trust proxy: true` / `*`, strips CORS headers, 1 MB body limit, token-bucket rate limit, dedupe cache.
- **YAML parsing** (`src/config/loader.ts:61`, `src/auth/policy.ts:243`) — `parseYaml(..., { maxAliasCount: 100 })` blocks billion-laughs-style aliasing.
- **Better-sqlite3 queries** — sampled across vault, dedupe, memory tiers; all use prepared statements with `?` placeholders. No string concatenation.
